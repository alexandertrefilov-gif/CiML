# 0004 Phase 1 Core Implementation Gate

Date: 2026-04-26

## Decision

Phase 1 may move from Skeleton/Foundation work into the first controlled Core implementation step.

The first allowed implementation step is limited to public-content CPT registration in `ciml-core` for:

- `ciml_belief_item`
- `ciml_ministry`
- `ciml_sermon`
- `ciml_event`
- `ciml_team_member`

This decision does not approve contact submissions, forms, admin pages, roles, capabilities, demo data installation, or Phase 2-5 features.

## Passed Checks

The following Skeleton/Foundation checks have passed:

- Theme presentation audit completed in `docs/qa/phase-1-theme-presentation-audit.md`.
- Plugin foundation audit completed in `docs/qa/phase-1-plugin-foundation-audit.md`.
- PHP syntax lint completed in `docs/qa/phase-1-php-lint-report.md`.
- PHPUnit foundation completed in `docs/qa/phase-1-phpunit-report.md`.
- PHP lint result: 29 PHP files checked, no syntax errors.
- PHPUnit result: 6 tests, 24 assertions, all passing.
- Theme remains presentation layer.
- Plugin remains core logic layer.
- Prototypes remain unchanged.
- No Phase 2-5 features are active.
- No sensitive modules are active.
- No forms or form processing are active.

## Tooling Boundary

`vendor/` and `.phpunit.cache/` are local tooling artifacts generated by Composer/PHPUnit.

Current status:

- they are not committed;
- they must not be added in the Phase 1 Core implementation commits;
- future `.gitignore` hardening should explicitly ignore them in a separate maintenance step.

This decision does not change `.gitignore`, because this step is documentation-only.

## Continuing Boundaries

The following boundaries remain active:

- Do not change `prototypes/`.
- Do not redesign the theme.
- Do not put business logic in `ciml-theme`.
- Do not implement Phase 2-5 features.
- Do not implement sensitive modules without a separate security gate.
- Do not claim production readiness.
- Do not invent real production content, legal text, contact data, or bank data.

## First Allowed Core Function

The first allowed Core implementation is public-content CPT registration only.

Allowed CPTs:

- `ciml_belief_item`
- `ciml_ministry`
- `ciml_sermon`
- `ciml_event`
- `ciml_team_member`

Allowed behavior:

- register the CPTs in `ciml-core`;
- keep them limited to Phase 1 public content;
- keep output ownership in the theme;
- add or update tests that verify registration code structure and safe boundaries;
- run PHP lint and PHPUnit after implementation.

No frontend rendering change is approved by this decision.

## Still Forbidden

The following remain forbidden until separately approved:

- `ciml_contact_message`
- contact forms
- form handling
- private submission storage
- email routing
- Admin Pages
- role registration
- capability registration
- demo data installer
- real options/settings pages
- taxonomies, unless approved in a separate small step after CPT registration
- public REST exposure decisions beyond the Phase 1 spec
- Phase 2 workflows such as smallgroups, courses, signups, volunteer applications, testimonies, or prayer workflows
- Phase 3 sensitive workflows such as seelsorge/care requests or confidential prayer data
- members area, rosters, internal documents, payments, governance, incident, or backup workflows

## Rationale

Public-content CPT registration is the lowest-risk first Core implementation step because:

- it models only public Phase 1 content already visible in the prototype;
- it does not process visitor input;
- it does not store sensitive data;
- it does not require new roles or capabilities yet;
- it does not require admin settings pages yet;
- it keeps the theme as presentation layer and the plugin as data/model layer;
- it can be tested structurally with the existing Composer/PHPUnit foundation;
- it creates the base for later editable public content without activating contact, admin, or sensitive workflows.

Contact submissions are intentionally excluded because they involve privacy, retention, routing, spam protection, and capability decisions that remain open.

## Required Checks After First Core Implementation

After the first CPT implementation step:

1. Run PHP lint for theme and plugin PHP files.
2. Run `vendor/bin/phpunit`.
3. Confirm `prototypes/` are unchanged.
4. Confirm `ciml-theme` has no CPT, taxonomy, role, admin, or form logic.
5. Confirm no Phase 2-5 features or sensitive modules were introduced.
6. Document the result in `docs/qa/`.

## Next Recommended Step

Implement only the five approved public-content CPT registrations in `ciml-core`, with focused tests and no admin UI, forms, roles, capabilities, demo data, or theme changes.
